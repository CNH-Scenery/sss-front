# 프론트-백엔드 정보 교환 방식

작성일: 2026-06-28  
대상: Tacit Trader 프론트엔드 ↔ CoinTwin FastAPI 백엔드

## 1. 기본 연결

- 백엔드 기본 API prefix: `/api`
- 프론트 개발 서버: Vite 기본값은 `http://localhost:5173`
- 백엔드 CORS 기본 허용값은 `http://localhost:3000,http://127.0.0.1:3000`이다.
- 프론트가 Vite 기본 포트를 쓰면 백엔드 실행 환경에 아래 값을 추가해야 한다.

```env
CORS_ALLOWED_ORIGINS=http://localhost:5173,http://127.0.0.1:5173,http://localhost:3000,http://127.0.0.1:3000
```

프론트에서는 API 기본 URL을 하나로 관리한다.

```js
const API_BASE_URL = import.meta.env.VITE_API_BASE_URL ?? "http://localhost:8000";
```

## 2. 인증 토큰 규칙

백엔드는 로그인/회원가입 성공 시 JWT access token을 내려준다.

프론트는 성공 응답의 `access_token`을 저장하고, 인증이 필요한 API에는 아래 헤더를 붙인다.

```http
Authorization: Bearer <access_token>
Content-Type: application/json
```

현재 백엔드 코드 기준으로는 `/api/auth/me`만 실제 토큰 인증을 사용한다. `/api/responses`, `/api/twin-contexts`, `/api/strategies`, `/api/trading-code`는 현재 `seed_anonymous_user()` 기반이라 로그인 사용자와 연결되지 않는다. 사용자별 데이터로 저장하려면 해당 라우터에서 `seed_anonymous_user()` 대신 `get_current_user`를 사용하도록 백엔드를 수정해야 한다.

## 3. 로그인

### 프론트 → 백

```http
POST /api/auth/login
Content-Type: application/json
```

```json
{
  "email": "trader@example.com",
  "password": "supersecret"
}
```

### 백 → 프론트

```json
{
  "access_token": "jwt-token",
  "token_type": "bearer",
  "user": {
    "id": "uuid",
    "name": "홍길동",
    "email": "trader@example.com"
  }
}
```

프론트 저장 권장값:

```json
{
  "accessToken": "jwt-token",
  "tokenType": "bearer",
  "user": {
    "id": "uuid",
    "name": "홍길동",
    "email": "trader@example.com"
  }
}
```

## 4. 회원가입

현재 프론트에는 회원가입 화면이 있으므로 백엔드의 회원가입 API도 같이 연결한다.

### 프론트 → 백

```http
POST /api/auth/signup
Content-Type: application/json
```

```json
{
  "name": "홍길동",
  "email": "trader@example.com",
  "password": "supersecret"
}
```

### 백 → 프론트

로그인과 같은 `AuthTokenResponse`를 받는다.

## 5. 로그아웃

현재 백엔드에는 로그아웃 endpoint가 없다. JWT 서버 세션을 저장하지 않는 구조라면 프론트에서 토큰을 삭제하는 방식으로 처리한다.

### 프론트 처리

```js
localStorage.removeItem("tt_session");
```

### 프론트 → 백

현재는 보낼 요청 없음.

### 백 → 프론트

현재는 응답 없음.

추후 토큰 블랙리스트나 refresh token을 도입하면 아래 endpoint를 추가한다.

```http
POST /api/auth/logout
Authorization: Bearer <access_token>
```

```json
{
  "message": "logged out"
}
```

## 6. 설문 시나리오 가져오기

백엔드에는 설문 차트/상황을 내려주는 endpoint가 이미 있다.

### 프론트 → 백

```http
GET /api/scenarios
```

### 백 → 프론트

```json
{
  "items": [
    {
      "id": "00000000-0000-4000-8000-000000000001",
      "market": "KRW-BTC",
      "timeframe": "15m",
      "description": "거래량이 증가하며 직전 고점을 테스트하는 구간",
      "features_snapshot": {
        "volume_ratio_n": 1.7,
        "recent_high_breakout": 0.8
      },
      "chart_data": [
        { "t": 1, "close": 100 },
        { "t": 2, "close": 103 }
      ]
    }
  ]
}
```

프론트 현재 차트 데이터는 `{ o, h, l, c, v, t }` 형태이고, 백엔드 seed 데이터는 `{ t, close }` 예시만 들어 있다. 실제 차트를 백엔드 데이터로 그릴 계획이면 `chart_data`를 아래처럼 OHLCV로 통일하는 것을 권장한다.

```json
{
  "t": "2026-06-28 15:00",
  "open": 92000000,
  "high": 92500000,
  "low": 91800000,
  "close": 92300000,
  "volume": 1234.56
}
```

프론트 내부 키를 유지하고 싶으면 아래 매핑을 둔다.

- `open` → `o`
- `high` → `h`
- `low` → `l`
- `close` → `c`
- `volume` → `v`
- `t` → `t`

## 7. 설문 결과 저장

현재 백엔드에 이미 있는 저장 endpoint는 `/api/responses`이다.

### 현재 코드로 바로 가능한 방식

프론트는 백엔드에서 받은 `scenario_id`를 기준으로 사용자의 판단만 보낸다. 설문용 차트 데이터와 설문 목적은 이미 `Scenario`에 저장되어 있으므로 중복 전송하지 않는다.

```http
POST /api/responses
Content-Type: application/json
Authorization: Bearer <access_token>
```

```json
{
  "scenario_id": "00000000-0000-4000-8000-000000000001",
  "decision": "buy",
  "natural_reason": "거래량이 터졌고 RSI가 과매도라 반등을 노리고 분할 매수한다.",
  "confidence": 0.72,
  "preferred_action": "split_buy"
}
```

허용되는 `decision` 값:

- 프론트 기본 3개: `buy`, `sell`, `hold`
- 백엔드 추가 허용값: `wait`, `add_position`, `take_profit`, `stop_loss`, `uncertain`

현재 프론트 버튼 값은 `BUY`, `SELL`, `HOLD`이므로 전송 전에 소문자로 바꾼다.

```js
const decision = draftAction.toLowerCase(); // BUY -> buy
```

### 백 → 프론트

```json
{
  "response_id": "uuid",
  "response_count": 1,
  "can_generate_twin": false
}
```

`can_generate_twin`은 응답 10개 이상이면 `true`가 된다.

### 요청사항을 100% 반영하는 확장 방식

사용자가 말한 “설문용 차트 데이터, 해당 설문 목적, 매수/매도/홀딩 여부, 자연어 이유”를 프론트가 모두 보내야 한다면 백엔드 스키마 확장이 필요하다. 권장 request body는 아래와 같다.

```json
{
  "scenario_id": "front-local-1",
  "market": "KRW-BTC",
  "timeframe": "15m",
  "survey_title": "급락 후 과매도 반등",
  "survey_purpose": "손실 공포 vs 역추세 매수 성향",
  "survey_reason": "큰 하락과 거래량 폭증 뒤 첫 반등을 보여준다.",
  "chart_data": [
    {
      "t": "2026-06-28 15:00",
      "open": 92000000,
      "high": 92500000,
      "low": 91800000,
      "close": 92300000,
      "volume": 1234.56
    }
  ],
  "features_snapshot": {
    "rsi14": 32.5,
    "vol_ratio": 1.7,
    "ma_align": "역배열"
  },
  "decision": "buy",
  "natural_reason": "RSI가 낮고 거래량이 붙어서 반등 가능성이 있다고 봤다.",
  "confidence": 0.72,
  "preferred_action": "split_buy"
}
```

이 확장 방식은 현재 `/api/responses` 스키마에는 맞지 않는다. 백엔드에서 새 endpoint를 만들거나 `UserResponseCreate`에 필드를 추가해야 한다.

권장 endpoint:

```http
POST /api/survey-results
```

## 8. 설문 응답 기반 코드 생성 흐름

백엔드의 현재 구조는 “응답 10개 → TwinContext 생성 → Strategy 생성” 흐름과 “자연어 prompt → 검증된 코인 코드 생성” 흐름이 따로 있다.

### 8-1. 설문 응답으로 성향 컨텍스트 생성

```http
POST /api/twin-contexts/generate
Authorization: Bearer <access_token>
```

request body 없음.

응답:

```json
{
  "context_id": "uuid",
  "version": 1,
  "style_summary": "거래량과 돌파 신호를 확인한 뒤 제한적으로 진입하는 중간 확신도의 성향입니다.",
  "important_signals": ["거래량 증가", "직전 고점 돌파"],
  "avoid_conditions": ["윗꼬리가 반복되는 구간"],
  "uncertainty": ["응답 10개는 TwinContext 생성을 위한 최소 표본입니다."]
}
```

### 8-2. 성향 컨텍스트로 전략 요약 생성

```http
POST /api/strategies/generate
Content-Type: application/json
Authorization: Bearer <access_token>
```

```json
{
  "context_id": "uuid"
}
```

응답:

```json
{
  "strategy_id": "uuid",
  "version": 1,
  "strategy_name": "CoinTwin Confirmation Strategy",
  "entry_threshold": 0.65,
  "summary": "거래량과 돌파 신호를 확인한 뒤 제한적으로 진입합니다."
}
```

주의: 이 endpoint는 현재 Python 코드 원문을 내려주지 않는다. 코드 원문이 필요하면 `/api/trading-code/generate`를 써야 한다.

## 9. 전략 정제/재코드화

프론트의 “재코드화” 입력창에서 사용자가 쓴 자연어 문장은 백엔드의 코드 생성 endpoint로 보낸다.

### 프론트 → 백

```http
POST /api/trading-code/generate
Content-Type: application/json
Authorization: Bearer <access_token>
```

초기 코드화 prompt 예시:

```json
{
  "prompt": "설문 응답 10개를 바탕으로 KRW-BTC 15분봉 전략을 만들어줘. 사용자는 RSI 과매도와 거래량 증가를 함께 볼 때 분할 매수를 선호하고, 윗꼬리와 거래량 둔화가 보이면 관망하거나 매도한다.",
  "market": "KRW-BTC",
  "timeframe": "15m",
  "max_iterations": 4
}
```

재코드화 prompt 예시:

```json
{
  "prompt": "기존 전략을 수정해줘. 손절을 더 빠르게 잡고, 거래량 조건은 더 엄격하게 해줘.",
  "market": "KRW-BTC",
  "timeframe": "15m",
  "max_iterations": 4
}
```

백엔드가 생성하는 코드는 아래 entrypoint를 만족해야 한다.

```python
def generate_signal(market: str, timeframe: str = "15m") -> dict:
    ...
```

반환 dict의 `action`은 반드시 소문자 `buy`, `sell`, `hold` 중 하나여야 한다.

### 백 → 프론트

```json
{
  "code_id": "uuid",
  "status": "passed",
  "passed": true,
  "iterations": 2,
  "model_name": "claude-model-name",
  "code": "def generate_signal(market: str, timeframe: str = \"15m\") -> dict:\n    ...",
  "verification": {
    "passed": true,
    "stage": "runtime",
    "errors": []
  },
  "decision_sample": {
    "action": "hold",
    "size_ratio": 0,
    "reason": "insufficient signal",
    "indicators": {}
  }
}
```

프론트는 `passed === true`이면 `code`를 코드 패널에 표시한다. `passed === false`이면 `verification.errors`를 사용자에게 보여준다.

## 10. 최신 생성 코드 조회

```http
GET /api/trading-code/latest
Authorization: Bearer <access_token>
```

응답은 `CodeGenerateResponse`와 같다. 아직 생성 코드가 없으면 `404`가 온다.

특정 코드 조회:

```http
GET /api/trading-code/{code_id}
Authorization: Bearer <access_token>
```

## 11. 프론트 구현 체크리스트

- 목 인증(`src/auth.js`)을 `/api/auth/signup`, `/api/auth/login`, `/api/auth/me` 호출 방식으로 교체한다.
- 로그인 성공 시 `access_token`, `token_type`, `user`를 저장한다.
- 로그아웃은 현재 로컬 토큰 삭제로 처리한다.
- 백엔드 API 호출마다 필요 시 `Authorization: Bearer ...`를 붙인다.
- 설문 시나리오는 가능하면 `/api/scenarios`에서 받아온다.
- 프론트 자체 생성 설문을 계속 쓸 경우, 백엔드에 설문 차트/목적까지 받는 확장 endpoint가 필요하다.
- 설문 응답 저장 시 `BUY/SELL/HOLD`를 `buy/sell/hold`로 변환한다.
- 코드화 버튼은 `/api/trading-code/generate`를 호출하고, 응답의 `code`를 표시한다.
- 재코드화 버튼도 같은 endpoint에 사용자의 수정 자연어를 `prompt`로 보낸다.

## 12. 백엔드 구현 체크리스트

- `/api/responses`, `/api/twin-contexts`, `/api/strategies`, `/api/trading-code`를 로그인 사용자 기준으로 저장하려면 `get_current_user`를 적용한다.
- Vite 포트 `5173`을 CORS 허용 목록에 추가한다.
- 설문 차트 데이터를 프론트의 OHLCV 차트가 바로 쓸 수 있는 형태로 확장한다.
- 설문 목적(`survey_purpose` 또는 `intent`)을 `Scenario` 모델에 추가할지 결정한다.
- 프론트가 설문 전체를 보내는 구조라면 `/api/survey-results` 같은 확장 endpoint를 추가한다.
- 전략 요약이 아니라 실제 Python 코드가 필요한 화면은 `/api/trading-code/generate`를 기준 계약으로 유지한다.
